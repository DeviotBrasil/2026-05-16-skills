# Referência de frameworks de ideação

Use estes frameworks de forma seletiva. Escolha a lente que casa com a ideia — não rode todo framework mecanicamente. O objetivo é destravar o pensamento, não seguir um checklist.

## SCAMPER

Forma estruturada de transformar uma ideia existente aplicando sete operações:

- **Substituir:** Que componente, material ou processo poderia ser trocado? E se substituíssem a tecnologia central? O público-alvo? O modelo de negócio?
- **Combinar:** E se uníssemos isto a outro produto, serviço ou ideia? Que duas coisas que normalmente não andam juntas criariam algo novo?
- **Adaptar:** O que mais é parecido com isto? Que ideias de outros setores, domínios ou épocas poderíamos emprestar? Que paralelo existe na natureza?
- **Modificar (Ampliar/Minimizar):** E se fosse 10× maior? 10× menor? E se exagerássemos um recurso? E se reduzíssemos ao mínimo absoluto?
- **Dar outros usos:** Quem mais poderia usar isto? Que outros problemas poderia resolver? E se fosse usado num contexto completamente diferente?
- **Eliminar:** E se removêssemos um recurso inteiro? Qual é a versão com zero configuração? Como seria com metade dos passos?
- **Inverter/Reorganizar:** E se fizéssemos os passos na ordem inversa? E se o usuário fizesse o trabalho em vez do sistema (ou o contrário)? E se invertêssemos a cadeia de valor?

**Melhor para:** Melhorar ou reimaginar produtos/recursos existentes. Menos útil para ideias greenfield.

## How Might We (HMW)

Reenquadre problemas como oportunidades com o formato "How Might We…":

- Comece com uma observação ou dor
- Reenquadre como "How might we [resultado desejado] para [usuário específico] sem [restrição-chave]?"
- Gere múltiplos enquadramentos HMW do mesmo problema — enquadramentos diferentes destravam soluções diferentes

**Boas qualidades de HMW:**
- Estreito o bastante para ser acionável ("…ajudar novos usuários a achar conteúdo relevante nos primeiros 5 minutos")
- Amplo o bastante para soluções criativas (não "…adicionar barra lateral de recomendações")
- Contém tensão ou restrição que força criatividade

**Más qualidades de HMW:**
- Largo demais: "How might we fazer usuários felizes?"
- Estreito demais: "How might we adicionar um botão na página de configurações?"
- Solução embutida: "How might we construir um chatbot para suporte?"

**Melhor para:** Reenquadrar pensamento travado. Quando alguém está preso a uma solução, puxe de volta ao problema.

## Pensamento de primeiros princípios

Quebre a ideia até verdades fundamentais e reconstrua a partir daí:

1. **O que sabemos ser verdade?** (não suposto, não convencional — de fato verdade)
2. **O que estamos supondo?** Liste toda suposição, inclusive as que parecem óbvias
3. **Quais suposições podemos contestar?** Para cada uma: "Isso é lei da física ou só 'sempre foi assim'?"
4. **Reconstrua a partir das verdades.** Se só tivesse as verdades fundamentais, o que construiria?

**Melhor para:** Sair do incremental. Quando toda ideia parece pequena melhoria sobre o status quo.

## Jobs to Be Done (JTBD)

Foque no que o usuário quer realizar, não no que diz querer:

- **Job funcional:** Que tarefa está tentando completar?
- **Job emocional:** Como quer se sentir?
- **Job social:** Como quer ser percebido?

Formato: "When I [situação], I want to [motivação], so I can [resultado esperado]."

**Insight-chave:** As pessoas não compram produtos — os "contratam" para fazer um job. O concorrente nem sempre é da mesma categoria. (Netflix compete com sono, não só com outros streamings.)

**Melhor para:** Entender o problema real. Quando você não tem certeza se está resolvendo a coisa certa.

## Ideação baseada em restrições

Imponha restrições de propósito para forçar soluções criativas:

- **Restrição de tempo:** "E se só tivesse 1 dia para construir isto?"
- **Restrição de recurso:** "E se só pudesse ter um recurso?"
- **Restrição de tech:** "E se não pudesse usar [a tecnologia óbvia]?"
- **Restrição de custo:** "E se tivesse que ser grátis para sempre?"
- **Restrição de público:** "E se seu usuário nunca tivesse usado computador?"
- **Restrição de escala:** "E se precisasse funcionar para 1 bilhão de usuários? E só para 10?"

**Melhor para:** Cortar complexidade. Quando a ideia cresceu grande demais ou vaga demais.

## Pré-mortem

Imagine que a ideia já falhou. Volte no tempo:

1. Daqui a 12 meses. O projeto foi lançado e fracassou. O que deu errado?
2. Liste todo motivo plausível de falha — técnico, mercado, equipe, timing
3. Para cada modo de falha: Isso é evitável? É sinal de que a ideia precisa mudar?
4. Quais modos de falha você aceita? Quais matariam o projeto?

**Melhor para:** Avaliação na Fase 2. Stress-testar ideias que parecem boas mas não foram pressionadas.

## Inspiração análoga

Veja como outros domínios resolveram problemas parecidos:

- Que indústria já resolveu uma versão deste problema?
- Como seria isto se [empresa/produto específico] construísse?
- Que sistema natural funciona assim?
- Que precedente histórico existe?

A chave é achar *estrutural* similaridade, não superficial. "Uber para X" é superficial. "Um mercado bilateral que resolve problema de confiança entre desconhecidos" é estrutural.

**Melhor para:** Expansão na Fase 1. Gerar variações que parecem genuinamente diferentes da abordagem óbvia.
